> 출처: input/질문/질문_v2.md (V-1, 실제 폼 문항). 답변 언어=한글, 폼 직접 입력.
> 생성 순서 1순위(E). 심사위원이 가장 집중해서 보는 부분.

## 실제 문항 목록 & 처리 전략

### ① 업스테이지 제품 체험 평가 (≤10,000자, 필수) ★★★ 핵심 중의 핵심
대상 제품: **Chat/Reasoning(Solar)**, **Document Digitization(Document Parse)**, **Information Extraction**.
체험 경로: Upstage Studio / Playground / Web·API (console.upstage.ai).

**이 문항의 본질** = JD의 "제품 피드백(재현절차·심각도 포함 QA 리포트 + mini-PRD)" 업무를 미리 시키는 것. → **답변의 형식 자체로 DevRel 역량을 증명**한다.

**답안 구조(안):**
1. **한 줄 평 + 체험 맥락(1문단)**: 어떤 실제 유스케이스로 써봤는지. (예: BookToss에 Document Parse로 '책 사진→제목 추출' 붙이기 / 비전공자용 교육 콘텐츠 생성)
2. **제품별 체험 — 각각 [무엇을 시켰나 → 좋았던 점 → 막혔던 점(재현 절차·심각도) → 비전공자/교육 관점]**
   - Chat/Reasoning (Solar)
   - Document Digitization (Document Parse)
   - Information Extraction
3. **종합 개선 제안 (mini-PRD 맛, 1~2개)**: 교육/DevRel 관점에서 임팩트 큰 개선.
4. **마무리**: 이 피드백을 쿡북/튜토리얼 콘텐츠로 확장하면 어떤 모습일지(이 직무에서 하고 싶은 일과 연결).

**원칙**: 구체적 입력/출력 예시, 재현 절차, 심각도(High/Med/Low) 표기. 칭찬·비판 균형. 솔직(안티 할루시네이션 — 안 되는 건 안 된다고). 10,000자 다 채울 필요는 없고 밀도 우선(목표 2,500~4,000자).
**[입력 필요]**: 실제 체험 결과(API 키 확보 후). 지금은 구조·체크리스트만. 06 데모 빌드와 동시에 채워짐.
