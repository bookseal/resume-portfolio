2. 배포 스크립트 2개

하나는 저장소 안(ops/deploy.sh, 실제 빌드/롤아웃), 하나는 서버 홈 (paste-jd-deploy.sh, git 갱신 후 앞의 것 호출).

ops/deploy.shbash (저장소에 커밋)
#!/usr/bin/env bash
set -euo pipefail
cd "$(dirname "$0")/.."   # 저장소 루트

echo "==> building image on server (arm64)"
docker build -t localhost:5000/paste-jd:latest web/
docker push localhost:5000/paste-jd:latest

echo "==> applying k8s manifests"
sudo -n k3s kubectl apply -f k8s/

echo "==> rolling out"
sudo -n k3s kubectl rollout restart deploy/paste-jd
sudo -n k3s kubectl rollout status deploy/paste-jd --timeout=180s
echo "==> done"
터미널 · ssh bit-habit — 서버 홈에 직접 작성
cat > ~/workspace/paste-jd-deploy.sh <<'EOF'
#!/usr/bin/env bash
set -euo pipefail
cd /home/ubuntu/workspace/resume-pipeline
before=$(git rev-parse HEAD)
git fetch --quiet origin main
git reset --hard origin/main
exec ops/deploy.sh "$before"
EOF
chmod +x ~/workspace/paste-jd-deploy.sh
저장소 쪽 스크립트 실행권한

로컬에서 chmod +x ops/deploy.sh 후 커밋하세요. git이 실행 비트를 보존합니다.

3. forced-command 배포 키

a

키 쌍 생성 (로컬)

터미널 · 로컬(저장소 밖 임시 위치)
ssh-keygen -t ed25519 -f /tmp/paste-jd-ci -N "" -C "paste-jd-ci-deploy"
cat /tmp/paste-jd-ci.pub    # 공개키 (서버에 등록)
cat /tmp/paste-jd-ci        # 개인키 (GitHub secret에 등록)
b

서버 authorized_keys에 forced-command로 등록

ssh bit-habit 후, 아래 한 줄을 ~/.ssh/authorized_keys에 추가합니다. ssh-ed25519 AAAA… 부분을 방금 출력된 공개키로 교체하세요.

~/.ssh/authorized_keys (서버) — 한 줄
restrict,command="/home/ubuntu/workspace/paste-jd-deploy.sh" ssh-ed25519 AAAA...공개키... paste-jd-ci-deploy
restrict + command

restrict는 포트포워딩·에이전트 등 모든 부가기능을 끄고, command="…"는 이 키로 들어온 어떤 요청이든 그 스크립트만 실행하게 강제합니다. 그래서 워크플로우가 보내는 true는 무시되고 배포 스크립트가 돕니다.

c

개인키를 GitHub secret으로

터미널 · 로컬 저장소
gh secret set DEPLOY_SSH_KEY < /tmp/paste-jd-ci
rm /tmp/paste-jd-ci /tmp/paste-jd-ci.pub   # 로컬 키 삭제

4. GitHub Actions 워크플로우

터미널 · 저장소 루트
mkdir -p .github/workflows
.github/workflows/deploy.ymlyaml
name: deploy
on:
  push:
    branches: [main]
    paths: ["web/**", "k8s/**", "ops/**", ".github/workflows/deploy.yml"]
  workflow_dispatch:

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Trigger server deploy over SSH
        run: |
          mkdir -p ~/.ssh
          echo "${{ secrets.DEPLOY_SSH_KEY }}" > ~/.ssh/id_ed25519
          chmod 600 ~/.ssh/id_ed25519
          ssh -o StrictHostKeyChecking=accept-new -i ~/.ssh/id_ed25519 \
            ubuntu@158.180.71.122 true
true의 정체

워크플로우는 true라는 무의미한 명령을 보냅니다. forced-command 때문에 서버는 이를 버리고 paste-jd-deploy.sh를 실행합니다 — CI 쪽엔 어떤 배포 로직도 노출되지 않습니다.

5. 첫 배포

먼저 PR을 merge해 main을 갱신하고, 첫 배포는 수동으로 안전하게 돌려봅니다.

터미널 · 로컬
chmod +x ops/deploy.sh
git add ops .github/workflows/deploy.yml
git commit -m "ci: SSH deploy pipeline + ops/deploy.sh (#9)"
git push -u origin feat/9-cicd
gh pr create --base main --title "ci: deploy pipeline (#9)" --body "Closes #9"
gh pr merge --squash --delete-branch
터미널 · 첫 배포 수동 실행 (서버)
ssh bit-habit '/home/ubuntu/workspace/paste-jd-deploy.sh'

이후부터는 main에 push만 하면 Actions가 자동으로 같은 스크립트를 돌립니다. 진행 상황은 gh run watch 또는 GitHub Actions 탭에서 확인.
