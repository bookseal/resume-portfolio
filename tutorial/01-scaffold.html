1. 이슈와 보드 만들기

a

gh CLI에 project 권한 추가

GitHub Projects(v2)를 CLI로 다루려면 project 스코프가 필요합니다. 한 번만 하면 됩니다.

터미널
gh auth status
gh auth refresh -s project,read:project
확인

gh auth status가 로그인 계정(bookseal)을 보여줘야 합니다. 아니면 gh auth login 먼저.

이 명령이 뭐 하는 걸까

gh auth refresh는 이미 로그인된 토큰을 새로 발급받는 명령입니다(다시 로그인 X). -s는 --scopes의 축약으로, 토큰에 권한을 더 붙입니다. project는 GitHub Projects 읽기+쓰기, read:project는 읽기 전용 권한이에요. 기본 gh auth login 토큰에는 이 스코프가 없어서, 그대로 gh project list를 치면 “read:project 스코프가 없다”는 에러가 납니다. 한 번 승인해 두면 계속 저장됩니다.

🧠 왜 이 명령일까
gh auth login으로 이미 로그인했는데, 왜 gh auth refresh -s project를 또 해야 할까요?
핵심: GitHub 토큰은 “무엇을 할 수 있는가(스코프)”가 처음 발급 때 정해집니다. 로그인 자체는 됐지만 권한 범위가 좁아서 Projects를 못 건드리는 상태죠. refresh -s는 로그인을 갈아엎지 않고 이 토큰에 권한만 추가하는 최소 조치라, 이게 “될 수밖에 없는” 명령입니다.
b

Project 보드 생성

터미널
gh project create --owner "@me" --title "paste-jd SaaS"
gh project list --owner "@me"

list에 나온 번호를 기억해 두세요(예: 3). 보드는 웹에서 github.com/users/bookseal/projects 로도 열립니다. 기본 컬럼(Todo / In Progress / Done)에 필요하면 웹 UI에서 In Review 컬럼을 하나 추가하세요.

이슈 #1 생성 (보드에 자동 등록)

--project 플래그로 생성과 동시에 보드에 카드가 꽂힙니다.

터미널 · resume-pipeline 저장소 루트에서

  --title "#1 Scaffold: web/ Next.js 앱 + 보드 셋업" \

Why 이슈부터

코드보다 이슈를 먼저 만드는 습관은, 작업을 작게 정의하고 나중에 “왜 이렇게 했나”를 추적 가능하게 만듭니다. 포트폴리오에서 이 흔적이 곧 실력의 증거가 됩니다.

2. Next.js 앱 스캐폴딩

a

브랜치 파기

작업은 항상 main이 아닌 브랜치에서. 카드를 보드에서 In Progress로 옮기세요.

터미널 · 저장소 루트
git switch main
git pull
git switch -c feat/1-scaffold
b

create-next-app 실행

저장소 루트에서 실행하면 web/ 하위에 앱이 생성됩니다. 이미 git 저장소 안이라 create-next-app은 새 git init을 건너뜁니다(정상).

터미널 · 저장소 루트
npx create-next-app@latest web \
  --typescript \
  --tailwind \
  --app \
  --src-dir \
  --eslint \
  --import-alias "@/*" \
  --use-npm \
  --no-turbopack
각 플래그 의미

--typescript 타입 안전 · --tailwind 유틸리티 CSS · --app App Router(최신 방식) · --src-dir 소스를 src/로 모음 · --import-alias "@/*" @/components/…처럼 절대경로 import · --no-turbopack 초심자용으로 안정적인 webpack dev. 대화형 질문을 모두 플래그로 미리 답해서 자동 진행됩니다.

npx · npm · turbopack — 용어 정리

npm = Node Package Manager. 라이브러리를 설치·버전관리하고 npm run dev 같은 스크립트를 돌리는 “관리자”. npx = Node Package eXecute. 패키지를 전역 설치 없이 한 번만 실행하는 “실행기”. create-next-app 같은 1회성 생성기에 딱 맞습니다 — 프로젝트 만들고 나면 다시 쓸 일이 없으니까요. Turbopack은 Vercel이 만든 Rust 기반 최신 번들러(dev 서버를 빠르게)지만, 아직 환경에 따라 에지 케이스가 있어 초심자 튜토리얼에선 검증된 webpack 경로로 고정합니다.

🧠 왜 npx일까
왜 npm install -g create-next-app으로 깔지 않고 npx create-next-app으로 실행할까요?
핵심: @latest가 붙은 이유와도 이어집니다. 생성기는 일회용 도구예요. 전역 설치는 “한 번 쓰고 안 쓸 칼을 계속 서랍에 두는” 것과 같습니다. npx는 필요할 때 꺼내 최신으로 쓰고 치우니, 버전이 낡을 걱정이 없습니다.
🧠 왜 옵션을 다 붙일까
옵션 없이 npx create-next-app@latest web만 쳐도 앱은 만들어집니다. 그런데도 --typescript --tailwind --app …을 다 붙이는 진짜 이유는?
핵심: 옵션이 없으면 create-next-app은 “TypeScript 쓸래? Tailwind 쓸래?”를 하나씩 물어봅니다. 튜토리얼은 그 답을 미리 다 정해 둔 거예요. 그래서 누가 따라 하든 결과가 동일하고 — 재현 가능성 — 스크립트에 넣어 자동화하기도 좋습니다.
c

생성된 구조 살펴보기

web/ 주요 구조tree
web/
├── package.json          # 의존성 + 스크립트(dev/build/start)
├── next.config.ts        # Next.js 설정
├── tsconfig.json         # TypeScript 설정 (@/* alias 여기 있음)
├── postcss.config.mjs    # Tailwind 파이프라인
└── src/
    └── app/
        ├── layout.tsx    # 모든 페이지를 감싸는 최상위 레이아웃(html/body)
        ├── page.tsx      # "/" 경로 = 홈 페이지
        └── globals.css   # 전역 스타일 + Tailwind 지시문
App Router 핵심

src/app/ 안의 폴더 이름이 곧 URL 경로이고, 그 폴더의 page.tsx가 그 경로의 화면입니다. 예: src/app/cookbook/page.tsx → /cookbook. #3에서 이걸 직접 씁니다.

라우팅 테이블을 코드로 쓰는 대신, 폴더를 만드는 것이 곧 라우트를 만드는 것.
곁가지: @/ 절대경로와 “linter”라는 이름

--import-alias "@/*" — 프로젝트가 커지면 import Button from "../../../components/Button" 같은 상대경로 지옥이 옵니다. @/를 항상 src/ 루트로 매핑해 두면 파일을 어디로 옮겨도 @/components/Button은 안 깨집니다(설정은 tsconfig.json에 저장). C의 -Iinclude로 include 루트를 잡는 것과 같은 발상이에요.

--eslint / linter의 어원 — lint는 원래 “옷의 보풀”. 1978년 Bell Labs에서 C 코드의 잠재 버그를 걸러내려고 만든 유닉스 도구 이름이 lint였고, “코드의 보풀을 털어낸다”는 비유였습니다. ESLint는 그 계보의 JS/TS판 — -Wall -Wextra를 팀 규칙까지 얹어 훨씬 빡세게 돌리는 별도 도구라고 보면 됩니다.

🧠 폴더=라우트 감각
src/app/dashboard/settings/page.tsx 파일을 만들었습니다. 이 화면은 어떤 URL에서 열릴까요?
핵심: page.tsx는 URL에 들어가지 않는 예약된 파일명 — “여기가 렌더링할 페이지다”라는 신호일 뿐입니다. URL을 만드는 건 폴더 계층 전체죠. 그래서 라우팅을 바꾸려면 코드가 아니라 폴더를 옮기면 됩니다. 이게 “파일시스템 기반 라우터”가 될 수밖에 없는 이유입니다.

4. 커밋 & PR

a

커밋

터미널 · 저장소 루트
cd ..
git add web
git commit -m "feat: scaffold Next.js app in web/ (#1)"
(#1)은 무슨 마법일까 — 사실 마법 아님

Git 입장에서 (#1)은 그냥 메시지 속 글자일 뿐, 특별 처리는 없습니다. 다만 GitHub UI가 메시지의 #숫자를 이슈/PR로 링크해 줘서, 나중에 “이 커밋이 왜 생겼지?”를 클릭 한 번으로 추적합니다. 이슈를 자동으로 닫는 건 전혀 다른 문법이에요 — PR 본문의 Closes #1 / Fixes #1 같은 키워드가 있어야 merge 시 이슈가 닫히고 보드 카드가 Done으로 갑니다.

🧠 링크 vs 자동 닫기
이 커밋 메시지의 (#1)은 merge될 때 이슈 #1을 자동으로 닫아줄까요?
핵심: #1은 사람과 UI를 위한 참조 표식(추적용), Closes #1은 GitHub에게 내리는 명령(lifecycle 연결)입니다. 둘을 구분해야, 왜 커밋엔 (#1)을, PR 본문엔 Closes #1을 따로 쓰는지가 납득됩니다. 이름만 비슷할 뿐 하는 일이 다릅니다.
b

푸시 & PR 생성

PR 본문에 Closes #1을 넣으면, merge 시 이슈가 닫히며 보드 카드가 자동으로 Done으로 이동합니다.

터미널
git push -u origin feat/1-scaffold
gh pr create --base main --title "feat: scaffold Next.js app (#1)" \
  --body "Closes #1

- web/ 에 create-next-app(TS·App Router·Tailwind)
- npm run dev 로컬 확인 완료"
--base는 “어디로 합쳐달라”인가

base = 변경이 흡수될 쪽(merge 대상), head = 변경이 들어있는 쪽(지금 작업 브랜치). 그래서 --base main은 “feat/1-scaffold → main으로 가는 PR”이라는 선언입니다. 팀에서 Git-flow를 쓰면 --base develop으로 바꿔 기능들을 먼저 develop에 모으고, 나중에 develop → main 릴리스 PR을 따로 냅니다. 개인 프로젝트는 보통 main 하나로 충분합니다.

%%{init: {"gitGraph": {"mainBranchName": "main"}} }%%
gitGraph
   commit id: "이전 작업"
   branch feat/1-scaffold
   checkout feat/1-scaffold
   commit id: "scaffold (#1)"
   checkout main
   merge feat/1-scaffold tag: "PR: base=main"
head(feat/1-scaffold)의 변경이 base(main)로 흘러 들어갑니다. base를 develop으로 바꾸면 합류 지점만 달라질 뿐 그림은 같아요.
🧠 base의 의미
어느 날 gh pr create --base develop로 바꿔 실행했습니다. 무엇이 달라질까요?
핵심: PR은 언제나 head → base 한 방향입니다. head는 “내가 지금 선 브랜치”로 자동 결정되고, --base만 “받는 쪽”을 고릅니다. 그래서 브랜치 전략(main 단일 / develop 분리)이 바뀌어도 바꿀 건 --base 값 하나뿐 — 이게 base라는 옵션이 존재하는 이유입니다.
c

리뷰 & merge

보드에서 카드를 In Review로 옮깁니다. Claude Code와 함께 diff를 살펴본 뒤, 문제없으면 merge하세요.

터미널
gh pr merge --squash --delete-branch

완료 체크리스트

클릭해서 체크하세요. 진행 상태는 이 브라우저에 저장돼 새로고침해도 남습니다.

  • Project 보드가 생성됐고 #1 카드가 보인다
  • gh 토큰에 project 스코프를 추가했다
  • web/에 Next.js 앱이 있고 npm run dev가 뜬다
  • http://localhost:3000에서 기본 페이지를 눈으로 확인했다
  • feat/1-scaffold 브랜치에서 커밋하고 push했다
  • PR을 올리고(Closes #1) merge했다 (카드 → Done)
이번에 얻은 감각

“이슈 → 브랜치 → 코드 → PR → merge” 한 바퀴를 돌았습니다. 앞으로 9번 더 반복합니다. 반복이 지루해질 때쯤, 이 루프가 몸에 붙어 있을 겁니다 — 그게 실무 개발의 기본기입니다.
