이 튜토리얼은 당신이 모든 코드와 터미널 명령을 직접 타이핑하도록 설계됐습니다. Claude는 대신 코드를 치지 않습니다 — 각 페이지의 코드/명령을 복붙하며 따라 하면, GitHub 이슈 하나가 곧 기능 하나가 되어 완성됩니다.

1순위 목표

JavaScript · TypeScript · React · Next.js를 손으로 익히기. Vibe coding이 아니라 hands-on.

2순위 목표

“Next.js + AI agent system” 포트폴리오 + 취준생이 clone하는 boilerplate.

만드는 것

기존 Python 파이프라인을 감싸는 SaaS 대시보드/뷰어. 랜딩 + cookbook + 매직링크 로그인.

배포

paste-jd.bit-habit.com — physical-spark과 같은 bit-habit k3s 인프라.

🧭 이 튜토리얼 사용법

  1. 위 보드에서 #1부터 순서대로 한 페이지씩 엽니다.
  2. 각 페이지 맨 위의 gh issue create 명령으로 그 이슈를 먼저 생성합니다.
  3. 브랜치를 만들고, 페이지의 코드/명령을 직접 타이핑(또는 복붙)합니다.
  4. 동작을 확인하고 PR을 올립니다 → 보드에서 카드가 In Review로 이동합니다.
  5. Claude Code와 함께 리뷰하고 merge → 카드가 Done으로. 다음 이슈로.
Tip

코드 블록 오른쪽 위 Copy 버튼으로 복사할 수 있습니다. 하지만 학습이 목적이니, 가능하면 직접 타이핑하며 각 줄이 무슨 일을 하는지 곱씹어 보세요.

🧰 시작 전 준비물

  • Node.js 20+ 와 npm (node -v 로 확인)
  • git 과 GitHub CLI gh (gh auth status 로 로그인 확인)
  • 이 저장소 resume-pipeline 로컬 클론 (여기 web/ 를 추가합니다)
  • 배포용(M3): docker, ssh bit-habit 접속 권한, kubectl (서버에서 사용)
  • 참고 사이트: infra.bit-habit.com — 지금 인프라가 어떻게 도는지

🔁 GitHub Project 워크플로우

이번에 GitHub Projects(칸반)를 처음 써 봅니다. 큰 그림은 이렇습니다:

컬럼의미언제 옮기나
Todo아직 시작 안 한 이슈이슈 생성 직후 자동
In Progress지금 작업 중브랜치 파고 코딩 시작할 때
In ReviewPR 올려 리뷰 대기gh pr create 직후
Donemerge 완료PR merge 시 자동
Why 이렇게

이슈를 잘게 쪼개 하나씩 처리하면 (1) 각 PR이 작아 리뷰가 쉽고, (2) 매 merge마다 배포가 살아있는 상태로 유지되며, (3) 포트폴리오에 “문제 → 해결” 커밋 히스토리가 또렷하게 남습니다. 큰 SaaS를 한 번에 만드는 게 아니라, 작은 기능을 반복해 쌓아 올리는 감각을 기르는 게 이 프로젝트의 진짜 학습 포인트입니다.
