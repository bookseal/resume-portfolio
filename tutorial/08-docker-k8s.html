web/Dockerfiledockerfile
# ---- 1) deps + build ----
FROM node:20-alpine AS builder
WORKDIR /app

# better-sqlite3(네이티브) 컴파일용 도구
RUN apk add --no-cache python3 make g++

COPY package.json package-lock.json ./
RUN npm ci

COPY . .
RUN npm run build       # .next/standalone 생성

# ---- 2) runtime ----
FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production
ENV PORT=3000
ENV HOSTNAME=0.0.0.0

# standalone 실행본 + 정적 파일 + public 을 각각 복사(자동 아님!)
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
COPY --from=builder /app/public ./public

# SQLite 데이터 디렉토리(마운트 지점)
RUN mkdir -p /data
ENV PASTE_JD_DB=/data/paste-jd.db

EXPOSE 3000
CMD ["node", "server.js"]   # standalone이 만든 진입점
public 복사 잊지 말 것

COPY .../public ./public가 없으면 #4의 /examples/*.docx 다운로드와 fs 읽기가 배포에서 깨집니다. standalone은 이 두 폴더를 자동으로 안 담습니다.

로컬 docker에서 로그인은?

이미지가 NODE_ENV=production이라 쿠키에 Secure가 붙습니다. http://localhost는 https가 아니라 브라우저가 쿠키를 저장하지 않아 로그인 테스트는 실패가 정상입니다. 로그인은 dev(#5–#7)와 실제 https 배포(#9)에서 검증됩니다. 여기선 “뜨는지”만 확인.

3. Kubernetes 매니페스트

저장소 루트에 k8s/ 폴더를 만들고 3개 파일을 둡니다. physical-spark의 pr-auth(빌드 이미지 + hostPath SQLite) 패턴을 그대로 따릅니다.

터미널 · 저장소 루트
mkdir -p k8s
k8s/deployment.yamlyaml
apiVersion: apps/v1
kind: Deployment
metadata: { name: paste-jd, namespace: default }
spec:
  replicas: 1
  strategy: { type: Recreate }        # SQLite 단일 writer
  selector: { matchLabels: { app: paste-jd } }
  template:
    metadata: { labels: { app: paste-jd } }
    spec:
      containers:
        - name: paste-jd
          image: localhost:5000/paste-jd:latest
          imagePullPolicy: Always
          ports: [ { containerPort: 3000 } ]
          env:
            - { name: PASTE_JD_DB, value: "/data/paste-jd.db" }
            - { name: PASTE_JD_SITE, value: "https://paste-jd.bit-habit.com" }
            - { name: PASTE_JD_COOKIE_DOMAIN, value: ".bit-habit.com" }
            - { name: PASTE_JD_OWNER, value: "gichanlee@icloud.com" }
          envFrom:
            - secretRef: { name: pr-auth-email, optional: true }  # SMTP secret 재사용
          readinessProbe:
            httpGet: { path: /api/health, port: 3000 }
            initialDelaySeconds: 5
          volumeMounts:
            - { name: data, mountPath: /data }
      volumes:
        - name: data
          hostPath:
            path: /home/ubuntu/workspace/paste-jd-data
            type: DirectoryOrCreate
k8s/service.yamlyaml
apiVersion: v1
kind: Service
metadata: { name: paste-jd-svc, namespace: default }
spec:
  selector: { app: paste-jd }
  ports:
    - { port: 80, targetPort: 3000 }   # 외부 80 → 컨테이너 3000
k8s/ingress.yamlyaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: paste-jd
  namespace: default
  annotations:
    traefik.ingress.kubernetes.io/router.entrypoints: websecure
    traefik.ingress.kubernetes.io/router.tls: "true"
spec:
  ingressClassName: traefik
  rules:
    - host: paste-jd.bit-habit.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service: { name: paste-jd-svc, port: { number: 80 } }
  tls:
    - hosts: [ paste-jd.bit-habit.com ]
      secretName: tls-secret          # 기존 *.bit-habit.com 와일드카드 인증서 재사용
Why DNS/인증서가 필요 없나

*.bit-habit.com 와일드카드 DNS(Route53)와 와일드카드 TLS(tls-secret)가 이미 있어서, 새 서브도메인은 Ingress 규칙 한 개만 추가하면 즉시 HTTPS로 살아납니다. 새 인증서 발급도, DNS 레코드 추가도 없습니다.

모든 경로가 같은 서비스로

physical-spark은 /api/auth를 별도 FastAPI 서비스로 분리했지만, 우리는 auth도 Next.js 안에 있으므로 / 하나로 전부 paste-jd-svc에 보냅니다. 같은 origin이라 쿠키도 first-party로 유지됩니다.
