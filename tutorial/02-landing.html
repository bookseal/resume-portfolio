0. 왜 React인가? (코드 치기 전에)

웹 화면은 결국 HTML입니다. “똑같이 생긴 카드 3개”를 만드는 가장 순진한 방법은 같은 HTML을 손으로 3번 복사하는 것 — 이걸 하드코딩이라 부릅니다(값을 코드에 직접 박기).

😵 하드코딩 (같은 모양 3번) · 개념 예시html
<div class="card"><h3>INPUT</h3>    <p>당신이 해온 모든 것…</p></div>
<div class="card"><h3>FUNCTION</h3> <p>회사별 JD…</p></div>
<div class="card"><h3>OUTPUT</h3>   <p>맞춤 이력서…</p></div>

문제는 바뀔 때 드러납니다. 디자인을 살짝 고치려면 3군데를 똑같이 고쳐야 하고, 하나라도 빠뜨리면 어긋나요. 카드가 100개면 재앙입니다. React의 발상은 하나: “모양(틀)은 한 번, 내용(데이터)은 따로.” 그 재사용 틀이 컴포넌트 — 붕어빵 틀이라고 생각하세요. 틀은 하나, 반죽(데이터)에 따라 다른 붕어빵(화면)이 나옵니다.

🧠 왜 틀로 분리하나
카드 3개를 복붙으로 만들었는데, 나중에 카드 디자인을 바꿔야 하면?
핵심: 복붙된 코드는 서로 남남입니다. “변하는 것”이 있으면 복붙은 곧 유지보수 지옥. React가 “틀 하나로 여러 번 찍기”를 미는 이유가 이 고통을 없애기 위해서입니다.
🧠 컴포넌트란
“컴포넌트”를 가장 잘 설명한 비유는?
핵심: 컴포넌트는 “데이터를 받아 화면을 만들어내는 함수”입니다. 틀은 고정, 내용은 매번 다르게. 카드가 3개든 300개든 틀 코드는 한 벌이면 됩니다.

개념 체크 — 스스로 설명할 수 있으면 체크. (여기부터 진행 바가 채워집니다)

  • 하드코딩(복붙)의 문제를 한 문장으로 말할 수 있다
  • 컴포넌트 = 재사용 UI 틀(함수)이라고 이해했다
  • “모양은 한 번, 내용은 따로”가 왜 이득인지 안다
  • 데이터 → map → 컴포넌트 → 화면 흐름을 떠올릴 수 있다
  • 이슈 #2를 만들고 feat/2-landing 브랜치를 팠다

0.5 C 하던 사람을 위한 TypeScript (걸리는 것만)

당신은 이미 변수·타입·배열·함수 포인터를 압니다. 그래서 TypeScript에서 “어? 이건 뭐지” 싶은 낯선 것만 C와 나란히 놓습니다. 이 6가지를 넘으면 다음 섹션의 landing.ts가 술술 읽힙니다.

① 원시 타입 이름이 다르다

C의 int / long / float / double은 TS에서 전부 number 하나(내부적으로 64-bit 부동소수). C의 char*/문자열은 string(널 종단·수동 메모리 없음). C의 _Bool은 boolean.

🧠 number
TS의 number는 C의 무엇에 대응하나요?
핵심: TS엔 정수/실수 구분 타입이 없습니다. 1도 1.5도 number. 그래서 정수 오버플로·형변환 같은 C의 고민이 (대개) 사라지는 대신, 큰 정수는 bigint를 따로 씁니다.

② type = typedef struct … 인데 “구조적”이다

C: typedef struct { char *label; } Stage; → TS: type Stage = { label: string };. 큰 차이는 타입을 무엇으로 구분하느냐입니다. C는 이름(nominal) — 필드가 같아도 다른 struct면 다른 타입. TS는 모양(structural) — 필드 모양만 맞으면 같은 타입으로 받아줍니다(덕 타이핑).

🧠 구조적 타이핑
type Stage = { label: string }가 있을 때, { label: "INPUT" } 객체는?
핵심: “오리처럼 걷고 오리처럼 울면 오리다”(덕 타이핑). C의 명목 타이핑에 익숙하면 놀랍지만, 덕분에 { ... } 리터럴을 그냥 함수에 넘겨도 모양만 맞으면 통과합니다. 다음 섹션의 STAGES가 이걸 그대로 씁니다.
③ 타입 표기는 뒤에 콜론

C는 타입이 앞: int x;. TS는 이름 뒤에 콜론: let x: number;. 함수도 function f(a: string): number { … } — 인자 타입은 뒤, 반환 타입은 ) 뒤 콜론.

🧠 타입 주석 위치
const STAGES: Stage[] = [];에서 : Stage[]는 무엇인가요?
핵심: 이름 : 타입 = 값 순서입니다. 사실 TS는 []만 봐도 타입을 추론하지만, 명시하면 “나는 여기에 Stage만 넣을 거야”라는 의도가 코드에 박혀 실수를 조기에 잡습니다.
④ 배열은 T[]이고 동적이다

Stage[] = “Stage 원소들의 배열”. C 배열은 고정 크기 + 포인터지만, TS 배열은 객체이고 크기가 가변이며 .map()·.filter() 같은 메서드를 가집니다. 경계를 넘어 접근하면 세그폴트가 아니라 undefined가 나옵니다.

🧠 T[]
string[]과 Stage[]의 []가 뜻하는 것은?
핵심: []는 크기가 아니라 “배열”이라는 타입 연산자입니다(Array<Stage>와 동일). 크기는 런타임에 .length로 변합니다 — C의 고정 배열과 가장 다른 점.
⑤ 객체 리터럴은 값이다(일급)

C에서 { .label = "INPUT" }는 초기화 문법일 뿐입니다. TS/JS에서 { label: "INPUT" }는 그 자체가 값이라 변수에 넣고, 함수에 넘기고, 배열에 담을 수 있습니다. 그래서 “객체들을 배열에 쭉 나열”하는 STAGES = [ {…}, {…} ]가 자연스럽게 가능합니다.

🧠 일급 객체
STAGES = [ {key:"input", …}, {key:"output", …} ]처럼 배열 안에 객체를 나열할 수 있는 이유는?
핵심: C에서 struct 리터럴은 “초기화 자리”에서만 살지만, 여기선 { … }가 1급 시민입니다. 변수 대입·함수 인자·배열 원소·반환값 어디든 됩니다. 이 자유가 “데이터를 데이터로 다루는” React식 사고의 바닥입니다.
⑥ export/import = 진짜 모듈 (헤더 없음)

C는 #include가 텍스트 붙여넣기이고, 선언은 .h·정의는 .c로 나뉘고, 링커가 나중에 잇습니다. TS는 파일이 곧 모듈: export한 것만 밖에서 보이고, import { X } from "./file"로 가져옵니다. 헤더·전방선언·링커 단계가 없습니다.

🧠 import vs #include
TS의 import가 C의 #include와 가장 다른 점은?
핵심: #include "landing.h"는 그 파일 내용을 통째로 펼치지만, import { STAGES } from "@/lib/landing"는 “landing 모듈이 export한 STAGES만” 콕 집어 씁니다. 그래서 이름 충돌·중복 정의 걱정이 없습니다.
보너스: 화살표 함수 => = 이름 없는 함수(람다)

(stage) => stage.title는 “stage를 받아 title을 돌려주는 익명 함수”. C의 함수 포인터/콜백과 역할은 같지만, 인라인으로 쓰고 주변 변수를 캡처(클로저)합니다. 뒤 .map() 섹션에서 이걸 씁니다.

TS↔C 이해 체크 — 각 항목을 “C로 치면 뭐다”로 말할 수 있으면 체크.

  • number가 C의 int/float 등 숫자 전부를 담당함을 안다
  • string에는 널 종단·수동 메모리가 없다는 걸 안다
  • type이 typedef struct 자리임을 안다
  • TS는 모양(구조적)으로 타입을 판정한다는 걸 안다
  • 타입 표기가 이름 뒤 콜론(x: T)임을 안다
  • T[]가 “T의 배열”이고 크기 가변임을 안다
  • 경계 밖 접근이 세그폴트가 아니라 undefined임을 안다
  • 객체 리터럴 { }가 일급 값임을 안다
  • export/import가 헤더 없는 진짜 모듈임을 안다
  • =>가 익명 함수(람다/콜백)임을 안다

1. landing.ts를 한 줄씩 (뼈대 → 살 붙이기)

이제 실제 파일을 시니어처럼 만듭니다. 완성본을 붙여넣지 않고, 빈 뼈대에서 시작해 타입 → 필드 → 배열 → 원소 → export 순서로 조금씩 키웁니다. 각 단계에서 “지금 이 줄이 왜 필요한가”를 묻습니다.

1.0 먼저: 우리가 뭘 만드나 (코드 0줄)

화면엔 카드 3개가 같은 모양, 다른 내용으로 반복됩니다. 그러면 “카드 한 개가 가진 정보”를 먼저 종이에 적습니다: 짧은 라벨(INPUT), 제목, 설명, 예시 한 줄, 그리고 React가 뒤에서 요구할 고유 id. 이 목록이 곧 타입의 필드가 됩니다.

🧠 무엇을 먼저
코드를 치기 전에 시니어가 가장 먼저 정하는 것은?
핵심: “화면에서 반복되는 게 뭔가? 그 한 조각이 가진 정보는?”을 먼저 답하면, 타입·배열·컴포넌트가 자동으로 따라옵니다. 설계를 데이터에서 시작하는 습관이 핵심입니다.
  • 화면에서 반복되는 요소(카드)를 찾았다
  • 카드 한 개가 가진 정보를 종이에 나열했다
  • 그 목록이 곧 타입 필드가 됨을 이해했다

1.1 빈 파일 + 의도 주석

web/src/lib/ 폴더를 만들고 landing.ts를 주석 두 줄로만 시작합니다. 시니어는 코드를 치기 전에 “이 파일이 뭐 하는 애냐”를 먼저 고정합니다.

web/src/lib/landing.tstypescript
// 랜딩 파이프라인 3단 데이터.
// 화면(page.tsx)이 이 데이터를 읽어 카드로 그린다.
🧠 왜 주석부터
빈 파일에 의도 주석부터 쓰는 이유로 가장 알맞은 것은?
핵심: lib은 “library”—화면이 아니라 재사용 데이터/로직을 두는 곳입니다. app(화면)과 lib(데이터)를 나누는 것 자체가 “내용과 모양의 분리”를 폴더 구조로 실천하는 것.
  • web/src/lib/ 폴더를 만들었다
  • landing.ts 파일을 만들었다
  • 파일 역할을 의도 주석으로 적었다

1.2 타입 껍데기부터

필드는 아직 안 씁니다. “Stage라는 모양을 만들 거야”라는 선언만 둡니다. C의 typedef struct {} Stage;와 대응.

landing.ts (추가)typescript
export type Stage = {
  // 필드는 곧 채운다
};
왜 벌써 export?

이 타입은 곧 컴포넌트 파일에서도 씁니다(import type { Stage }). 밖에서 쓸 걸 알면 처음부터 export를 답니다.

🧠 빈 type
type Stage = {}; — 아직 필드가 없는 이 줄의 의미는?
핵심: type은 값이 아니라 약속입니다. 실행되지도, 메모리를 쓰지도 않아요. 컴파일 후엔 사라집니다(브라우저엔 JS만 남음). 지금은 “이름부터 세운다”는 뼈대 단계입니다.
  • export type Stage = { } 뼈대를 넣었다
  • type이 “값이 아니라 모양의 약속”임을 안다
  • 밖에서 쓸 것이라 export를 붙인 이유를 안다

1.3 첫 필드: 고유 id key

필드 하나만 넣습니다. 각 단을 구분할 고유 id. 이름을 key로 고른 건, 뒤에서 .map()의 React key로 쓸 것이기 때문입니다(4번 섹션에서 회수).

landing.ts (필드 추가)typescript
export type Stage = {
  key: string;        // "input" | "function" | "output" — 각 단의 고유 id
};
🧠 왜 key 필드
데이터 한 개에 고유 id(key)를 미리 넣어두는 이유는?
핵심: 설계를 미리 하면 “나중에 필요할 것”을 데이터 모양에 먼저 반영합니다. 이 key가 #1의 체크리스트 “고유 인덱스”와 같은 이유로 존재합니다 — 똑같이 생긴 게 여럿일 땐 신원표가 필요.
🧠 string
key: string의 string은 C의 무엇 자리인가요?
핵심: string은 진짜 문자열 타입입니다. len 계산에 strlen도, \0도, malloc도 필요 없습니다.
  • key: string; 필드를 넣었다
  • key가 나중 .map()의 React key로 쓰일 걸 안다
  • string이 C char* 자리(안전판)임을 안다

1.4 나머지 필드 채우기 (전부 텍스트)

화면에 보일 나머지 정보. 전부 텍스트라 타입은 모두 string. 세미콜론으로 줄을 끊습니다(쉼표도 허용).

landing.ts (필드 완성)typescript
export type Stage = {
  key: string;        // 고유 id (map key)
  label: string;      // INPUT / FUNCTION / OUTPUT — 작은 배지 텍스트
  title: string;      // 카드 제목
  body: string;       // 설명 한두 줄
  example: string;    // John Doe 예시 한 줄
};
🧠 다 string
다섯 필드의 타입이 전부 string인 이유는?
핵심: 타입은 “이 필드에 뭐가 들어올지”를 데이터의 성격에서 고릅니다. 나중에 “표시 순서” 같은 게 필요하면 order: number를, “강조 여부”면 featured: boolean을 섞겠죠.
🧠 구분자
type 안에서 필드를 끊는 ;(세미콜론)은?
핵심: 타입 정의에서 ;와 ,는 취향입니다. 팀 코드 스타일(그리고 ESLint/Prettier)이 하나로 통일해 줍니다.
  • label 필드를 넣었다
  • title 필드를 넣었다
  • body 필드를 넣었다
  • example 필드를 넣었다
  • 타입을 “데이터의 성격”에서 고른다는 감을 잡았다
  • 필드 구분자 ;/,가 취향임을 안다

1.5 배열 뼈대: 빈 STAGES

타입은 “한 개의 모양”이었습니다. 이제 “그 모양 여러 개”를 담을 배열을 빈 채로 선언합니다. 뼈대 먼저, 원소는 다음 단계에.

landing.ts (추가)typescript
export const STAGES: Stage[] = [];
//           ^^^^^^  ^^^^^^^   ^^
//           이름     타입     아직 빈 배열
🧠 Stage vs Stage[]
Stage와 Stage[]의 차이는?
핵심: “한 개의 타입”을 만들고 “그 배열”로 여러 개를 담는 이 패턴이 데이터 설계의 기본형입니다.
🧠 왜 빈 배열
원소를 바로 넣지 않고 = [] 빈 배열로 먼저 두는 이유는?
핵심: “작게 만들고 자주 확인”이 시니어의 리듬입니다. 뼈대가 초록불(에러 없음)이면, 그 위에 살을 붙일 때 문제 원인을 좁히기 쉽습니다.
🧠 const
const STAGES의 const는 배열 내용까지 얼리나요? (C의 const와 비교)
핵심: C의 const와 미묘하게 다릅니다. JS const는 “변수 바인딩”을 고정할 뿐, 가리키는 객체 내부는 아닙니다. 완전 불변이 필요하면 as const/readonly를 씁니다. 지금은 안 바꿀 거라 const로 충분.
  • export const STAGES: Stage[] = [];를 넣었다
  • Stage(한 개) vs Stage[](배열) 차이를 안다
  • “뼈대 먼저, 살 나중”의 이유를 안다
  • const가 재할당만 막는(얕은) 것임을 안다

배열에 첫 객체를 넣습니다. Stage의 다섯 필드를 다 채우면, 구조적 타이핑에 의해 “이건 Stage다”가 됩니다.

핵심: 0.5의 ②가 여기서 실전으로 옵니다. 이름표 없이 모양만 맞으면 통과 — C 명목 타이핑과 가장 다른 대목.
🧠 안전망 체감
실수로 example 필드를 빠뜨리면 어떻게 될까요?
핵심: 이게 JS 대비 TS의 최대 실무 이점입니다. “타입이라는 약속”을 어기는 순간 컴파일 전에 잡혀요. C의 컴파일 경고가 더 똑똑해지고 에디터 안에서 실시간으로 뜬다고 보면 됩니다.
  • INPUT 객체를 배열에 넣었다
  • 다섯 필드를 모두 채웠다
  • “모양이 맞으면 Stage”라는 감을 잡았다
  • 일부러 필드를 빼서 빨간 줄을 눈으로 확인했다

🧠 확장 비용
파이프라인에 4번째 단을 추가하고 싶다면 이제 무엇을 하나요?

  • 문자열 안 \" 이스케이프를 이해했다

1.8 마지막: 밖으로 열기 (export 확인)

우리는 type Stage와 const STAGES 둘 다에 이미 export를 붙였습니다. 이게 없으면 데이터는 이 파일 안에만 갇혀 컴포넌트가 못 씁니다. C로 치면 “.h에 공개 안 한 static 심볼”과 비슷.

완성된 landing.ts 골격typescript
export type Stage = { key; label; title; body; example: string };
export const STAGES: Stage[] = [ /* input, function, output */ ];
//    ^^^^^^ 이 두 export가 page.tsx / StageCard.tsx의 import를 가능하게 한다
🧠 export 없으면
STAGES에서 export를 빼면 어떻게 되나요?
핵심: 모듈은 기본이 비공개입니다. export한 것만 “공개 API”가 되죠. C에서 헤더에 선언을 노출해야 다른 .c가 쓸 수 있던 것과 같은 개념 — 단, 링커 없이 파일 단위로 자동입니다.
  • type Stage에 export가 있다
  • const STAGES에 export가 있다
  • 모듈이 기본 비공개이고 export가 공개임을 안다
  • landing.ts가 에러 없이 저장된다(초록불)

데이터가 준비됐으니 틀을 만듭니다. web/src/components/ 폴더에 한 단(stage)을 그리는 컴포넌트를 둡니다. 데이터는 props로 받습니다 — 함수의 인자라고 생각하세요.

JSX 규칙 3가지

① HTML의 class는 JSX에서 className. ② 중괄호 { } 안엔 JS 표현식({stage.title}). ③ 컴포넌트는 반드시 하나의 최상위 요소를 반환(여기선 바깥 <div>).

🧠 props란
props(여기선 { stage })에 가장 가까운 개념은?
핵심: StageCard({ stage })는 함수, stage는 인자. INPUT 데이터를 넣으면 INPUT 카드가, OUTPUT을 넣으면 OUTPUT 카드가 나옵니다. props가 곧 붕어빵 반죽입니다.
🧠 구조 분해
function StageCard({ stage })에서 인자 자리의 { stage }는?
핵심: props.stage라고 매번 쓰는 대신, 인자에서 바로 { stage }로 꺼냅니다. C엔 없는 편의 문법 — 객체를 “필요한 필드만” 풀어 받는다고 보면 됩니다.
🧠 : { stage: Stage }
인자 뒤의 : { stage: Stage }는 무엇을 하나요?
핵심: 이 덕에 <StageCard />를 stage 없이 쓰거나 엉뚱한 걸 넘기면 즉시 빨간 줄이 뜹니다. 컴포넌트도 “서명”이 있는 함수입니다.
🧠 class vs className
HTML은 class인데 JSX는 왜 className일까요?
핵심: JSX 태그는 눈엔 HTML 같지만 JS로 변환됩니다. “보이는 것 ≠ 실제”라는 걸 아는 게 JSX의 첫 관문입니다.
🧠 최상위 하나
컴포넌트가 하나의 최상위 요소를 반환해야 하는 이유에 가장 가까운 것은?
핵심: return은 값 하나만 돌려줍니다. 감쌀 div가 싫으면 빈 <>…</> Fragment를 씁니다.
  • web/src/components/ 폴더를 만들었다
  • StageCard.tsx를 만들었다
  • import type { Stage }로 타입을 가져왔다
  • props를 { stage }로 구조 분해했다
  • props 타입 { stage: Stage }를 명시했다
  • class가 아니라 className을 썼다
  • 중괄호 {stage.title}로 데이터를 꽂았다
  • 하나의 최상위 <div>로 감쌌다
  • export default로 컴포넌트를 열었다
  • “컴포넌트 = props를 받는 함수”라고 말할 수 있다

3. .map() 완전 분해 (C for 루프 ↔ map)

데이터(1)와 틀(2)이 있으니, 남은 건 “데이터 3개를 카드 3개로 변환”하는 것. C였다면 결과 배열을 잡고 for로 하나씩 채웠겠죠. 바로 그 패턴을 한 줄로 접은 게 .map()입니다.

C 방식: 손으로 루프 · 개념 예시c
Stage in[3]  = { /* input, function, output */ };
Card  out[3];
for (int i = 0; i < 3; i++) {
    out[i] = make_card(in[i]);   // 하나씩 변환해 새 배열에 채움
}
JS/TS 방식: 그 패턴을 .map()으로typescript
const out = STAGES.map((stage) => makeCard(stage));
//                      └── C의 make_card(콜백)에 해당하는 익명 함수 ──┘
// 결과: 길이 3짜리 "새 배열". STAGES 원본은 그대로.
flowchart LR
  A["[ a, b, c ]"]:::src --> F{{"map( f )"}}:::fn --> B["[ f(a), f(b), f(c) ]"]:::dst
  A -.->|"원본 불변"| A
  classDef src fill:#f59e0b,color:#141018,stroke:#d97706,stroke-width:2px
  classDef fn  fill:#8b5cf6,color:#fff,stroke:#7c3aed,stroke-width:2px
  classDef dst fill:#0e9a89,color:#fff,stroke:#0b7f70,stroke-width:2px
.map(f) = 각 원소에 f를 적용해 같은 길이의 새 배열을 만든다. 원본은 안 바뀐다.
map의 4가지 성질

① 새 배열을 반환한다(원본 불변). ② 길이를 보존한다(3개 → 3개). ③ 각 원소에 인자 함수 f를 적용한다. ④ 그 f는 익명 함수(=>)로 인라인 작성 — C 콜백을 매번 이름 붙여 선언할 필요가 없습니다.

🧠 반환값
STAGES.map(...)가 돌려주는 것은?
핵심: map은 “배열 → 배열” 변환기입니다. 그 반환 배열을 JSX에 그대로 꽂으면 React가 카드들을 렌더합니다.
🧠 원본은?
.map() 실행 후 원본 STAGES 배열은?
핵심: “원본 불변”이 map의 큰 미덕입니다. C에서 in[]을 안 건드리고 out[]만 채운 것과 같아요 — 예측 가능성이 올라갑니다.
🧠 길이
입력이 3개면 .map() 결과의 길이는?
핵심: 걸러내고 싶으면(길이 감소) .filter(), 하나로 합치고 싶으면 .reduce()가 따로 있습니다. map은 1:1 변환 전용.
🧠 (stage) => …
.map((stage) => …)의 (stage) => …는 C의 무엇과 가장 가깝나요?
핵심: C 함수 포인터의 진화형이라고 보면 됩니다. “make_card를 따로 선언 → 이름 넘기기” 대신, 변환 로직을 그 자리에 적습니다. 짧고, 문맥이 눈앞에 있습니다.
🧠 왜 for 대신 map
배열 변환에 손수 짠 for 루프보다 .map()이 나은 지점은?
핵심: 속도가 아니라 의도의 명료함과 실수 제거가 이유입니다. i < 3을 i <= 3으로 쓰는 류의 버그가 원천 봉쇄됩니다.
  • C의 “결과 배열 + for 변환” 패턴을 떠올릴 수 있다
  • .map()이 그 패턴을 한 줄로 접은 것임을 안다
  • map이 새 배열을 반환함을 안다
  • map이 원본을 안 바꾼다는 걸 안다
  • map이 길이를 보존(1:1)함을 안다
  • filter(감소)·reduce(합치기)와 역할이 다름을 안다
  • (stage) => …가 인라인 콜백(람다)임을 안다
  • for보다 map이 “의도가 드러나고 실수가 준다”를 이해했다

4. 홈 페이지 조립 (map 적용 + key 회수)

이제 데이터(1) + 틀(2) + map(3)을 합칩니다. web/src/app/page.tsx를 아래로 교체하세요. 1.3에서 심어둔 key가 여기서 회수됩니다.

web/src/app/page.tsxtsx

Why key가 필요한가 (회수)

리스트를 .map()으로 그릴 때 React는 각 항목을 구분할 고유 key가 필요합니다. 없으면 콘솔 경고가 뜨고, 항목이 바뀔 때 렌더가 꼬입니다. 그래서 1.3에서 데이터에 key를 미리 심어뒀고, 여기서 key={stage.key}로 씁니다.

🧠 중괄호 in JSX
JSX 안에서 {STAGES.map(...)}처럼 중괄호로 감싸는 이유는?
핵심: JSX는 “마크업 안에 {JS}를 섬처럼 끼우는” 문법입니다. {stage.title}, {STAGES.map(...)} 모두 그 섬입니다.
🧠 map 결과가 화면에
{STAGES.map((stage) => <StageCard stage={stage}/>)}가 실제로 만드는 것은?
핵심: 3번 섹션의 “map=배열→배열”이 그대로 화면이 됩니다. STAGES에 4번째를 넣으면 카드도 4개 — page.tsx는 손 안 대고.
🧠 key 회수
key={stage.key}가 필요한 이유는? (#1 체크리스트 인덱스 충돌과 연결)
핵심: #1에서 체크박스가 “고유 인덱스” 없이 겹쳐 함께 켜지던 버그와 똑같은 원리입니다. 똑같은 게 여러 개면 시스템엔 “신원”이 필요합니다. React의 key가 그 신원표예요.
🧠 import 3줄
page.tsx 맨 위 import { STAGES } from "@/lib/landing"가 하는 일은?
핵심: 0.5의 ⑥이 실전으로 옵니다. #include식 복붙이 아니라 “export된 이름만 콕” — 그리고 @/는 #1에서 세운 절대경로 별칭(src/ 기준)입니다.

6. 마무리 · 이해도 체크

코드가 아니라 머릿속에 남았는지를 확인하세요. 다 채우면 이 페이지가 100% 됩니다. 🎉

  • 하드코딩 vs 데이터 분리의 차이를 설명할 수 있다
  • C의 typedef struct ↔ TS type을 매핑할 수 있다
  • 구조적 타이핑(모양으로 판정)을 설명할 수 있다
  • 타입 표기가 뒤 콜론(x: T)임을 안다
  • Stage vs Stage[]를 구분할 수 있다
  • “뼈대 먼저, 살 나중”의 작업 순서를 설명할 수 있다
  • 컴포넌트와 props(함수와 인자)를 설명할 수 있다
  • .map()을 C for 루프로 바꿔 설명할 수 있다
  • key가 왜 필요한지 말할 수 있다
  • 데이터 → map → 컴포넌트 패턴을 다음 페이지에서도 알아볼 수 있다

화면을 “손으로 그리는 것”에서 “데이터로 서술하는 것”으로 사고가 넘어갔고, C의 무기(struct·배열·콜백)가 TS에서 어떻게 바뀌는지 지도를 얻었습니다. 이 데이터 → map → 컴포넌트 패턴은 #3·#4에서 똑같이 반복됩니다 — 이제 그게 눈에 보일 겁니다.
