② JD 원문 job description

[업스테이지는] AI 기술을 "진짜 제품"으로 숨쉬게 만들 분을 찾습니다. 복잡한 AI 파이프라인을 코드 한 줄 몰라도 드래그 앤 드롭으로 설계할 수 있는 세상, Agent 들이 편하게 가져다 쓸 수 있는 고성능의 스킬 — 사용자 경험의 정점을 찍는 인터페이스와 워크플로우를 함께 설계할 인턴 엔지니어. 우리 팀은 이런 문제를 풀고 있습니다 - 수천 건의 계약서에서 핵심 정보를 최고의 정확도로 추출하는 워크플로우 빌더 - Human-in-the-loop와 자동 평가를 결합해, 최소한의 고객 입력으로 품질이 빠르게 오르는 피드백 루프 - AI가 분석한 복잡한 추출 결과와 추론 과정을 모니터링하는 직관적인 대시보드 주요 업무 - AI 서비스 UI/UX 고도화 · 데이터 기반 기능 개발(모니터링 대시보드·시각화) - Cross-Functional: 모델 연구팀(최신 기술)·미국 팀(글로벌 시장)과 협업 자격 요건: TypeScript/Python + 웹 서비스 · AI 도구로 10x · 작은 기능도 끝까지 완성 우대: Next.js 풀스택 배포 · AI/ML·RAG · 데이터 시각화 · 영어

③ 이 일이 뭔가 — FDE와 뭐가 다른가

한 문장으로: "AI가 계약서에서 뽑은 정보를, 사람이 믿고 쓸 수 있게 만드는 프론트엔드 중심 제품 엔지니어." FDE처럼 보이지만 방향이 반대다.

① 계약서 정보 추출 워크플로우 → KIBA_수문 · RAG

KIBA_수문: 정부 지침 PDF 5개 + 엑셀 8개에서 예산·원가 정보를 추출·정제해 Streamlit 앱으로. 모든 숫자에 "원본 파일·시트·셀" 단위 citation이 붙는다 (validation-first). 원본 엑셀 뷰어로 직접 대조까지.
→ 이게 정확히 "수천 건 계약서에서 정보 추출 + 최고 정확도" 문제의 구조다. 문서 → 구조화 → 검증.

llm-app-lab RAG: 항공법 1,297쪽에서 §인용과 함께 답을 뽑는 RAG. 문서에서 정확한 정보를 뽑아내는 파이프라인 그 자체.

② Human-in-the-loop + 자동 평가 → pm_with_ai_agent · eval 하네스

project_management_with_ai_agent: 회의록 → AI 에이전트가 GitHub 보드 변경을 제안하되, "당신이 확인하기 전엔 아무것도 쓰지 않는다." 읽기는 자동, 쓰기는 항목별 사람 승인, 삭제·권한·코드는 금지. → 이게 "Human-in-the-loop"의 교과서적 구현.

RAG eval 하네스: 45개 설정을 blind 홀드아웃에 자동 채점(coverage/recall/MRR), §인용 실재 여부까지 프로그램 검증. → 이게 "자동 평가(auto-eval)". 사람 입력 최소로 품질을 재는 장치.

③ 추론 결과 신뢰 대시보드 → KIBA_수문 · quali-fit

KIBA_수문의 citation-first가 여기도 걸린다 — AI/데이터가 낸 숫자를 "어디서 나왔는지 원본으로 되짚을 수 있게" 만든 것. 이게 신뢰 대시보드의 본질(provenance).

quali-fit: "누구를, 왜 투입하는가"를 이유와 함께 점수로 보여주는 사내 웹 도구. 도메인 로직을 UI·DB에서 분리, 단위 테스트로 고정. → AI/알고리즘 결과를 비전문가가 신뢰하고 판단하게 만드는 UI. 정확히 이 팀의 3번 문제.

⑤ JD 문장별 해설 — 미국/선진 엔지니어링 문화 렌즈

Upstage는 미국식 AI 엔지니어링 문화(evals·신뢰성·high agency)를 따라간다. JD의 각 문구를 그 렌즈로 풀면, "왜 이 문장을 썼는지"가 보이고 내 경험이 어디에 꽂히는지 명확해진다.

"AI 기술을 '진짜 제품'으로 숨쉬게 만들 분"

US 문화 미국 AI 업계의 핵심 구분: research demo vs production product. 모델이 되는 것과 사람이 매일 쓰는 제품이 되는 것은 다르다. "숨쉬게(breathe)"는 살아서 운영된다는 뜻.
내 경험 나는 BookToss·quali-fit·KIBA 에이전트를 데모가 아니라 2년째 운영되는 것으로 만들었다. "데모에서 멈춘 것과 살아있는 것 사이의 거리를 걸어봤다"가 내 문장.

"코드 한 줄 몰라도 드래그 앤 드롭으로 AI 파이프라인 설계"

US 문화 abstraction for non-experts — 복잡한 걸 비전문가가 다루게 만드는 것이 미국 제품 문화의 최고 가치(Figma, Zapier, Retool 계보). "코드 없이"는 곧 사용자 층을 개발자 밖으로 넓힌다는 뜻.
내 경험 KIBA에서 "비전공자 팀원이 나 없이도 워크플로우를 돌릴 수 있게" 문서화·온보딩한 것이 정확히 이 가치. 자동화가 나에게만 의존하면 그건 병목이라는 관점.

"수천 건의 계약서에서 핵심 정보를 최고의 정확도로 추출"

US 문화 엔터프라이즈 AI의 핵심 난제 = accuracy at scale + trust. 계약서는 틀리면 법적/금전적 비용이 크다. 그래서 "최고의 정확도"는 곧 측정·검증 체계를 요구한다.
내 경험 KIBA_수문의 셀 단위 citation(validation-first) + RAG의 §인용 실재 여부 프로그램 검증. 나는 이미 "추출한 값이 맞는지 원본으로 되짚는" 시스템을 만들었다.

"Human-in-the-loop와 자동 평가를 결합"

US 문화 안전하게 AI를 배포하는 표준 패턴 두 개. HITL = 사람이 승인/교정, evals = 감이 아니라 지표로 품질 측정. Anthropic·OpenAI가 밀어붙이는 "evals over vibes" 문화.
내 경험 HITL은 pm_with_ai_agent(쓰기 전 사람 승인, 삭제 금지), auto-eval은 RAG 45-config 하네스. 둘을 각각 실제로 만들어봤다 — 이 문장은 내 두 프로젝트를 합친 것과 같다.

"AI가 분석한 추론 과정을 모니터링하는 직관적인 대시보드"

US 문화 observability / provenance — "이 답을 믿어도 되나?"에 답하는 UI. 결과만이 아니라 근거·과정을 보여줘 사용자가 스스로 신뢰 수준을 판단하게 한다.
내 경험 quali-fit의 "이유와 함께 추천", KIBA_수문의 원본 대조 뷰어. 사전질문 답안에도 "확신도 미표현을 UI 문제로 재해석"이라 썼다 — 이 팀의 3번 문제를 이미 그 각도로 봤다.

"AI 도구를 실무에 적극 활용해 10x로 개발 효율을 높일 분"

US 문화 2025~2026 미국 테크의 최신 규범: AI-native development. Claude Code/Cursor로 개발 속도를 몇 배로. "쓸 줄 안다"가 아니라 "이걸로 일하는 방식이 바뀌었나"를 본다.
내 경험 현직에서 Claude Code·Codex를 개발 프로세스 자체에 이식 — 이슈→PR→리뷰 자동화, 지침서 검토 10h→2h. 이건 나의 최강 카드. 메일이 "AI 도구 준비"를 요청한 이유이기도 하다.

"작은 기능 하나라도 끝까지 책임지고 완성 · high agency · 플레이북이 아직 쓰이는 환경"

US 문화 스타트업의 ownership / high agency — 시키는 걸 하는 게 아니라 문제를 발굴하고 끝까지 끈다. "플레이북이 아직 쓰이는 중"은 정답이 없는 환경에서 스스로 길을 낸다는 뜻.
내 경험 나는 기획→개발→배포→운영을 혼자 완주한 서비스가 14개. 창업 경험(4년)까지. "혼자 다 해봤다"가 이 환경에 대한 증거.

"Cross-Functional: 모델 연구팀 · 미국 팀과 협업"

US 문화 research ↔ product bridge, 그리고 글로벌 협업(영어·비동기). 미국 팀과 붙는다는 건 이 제품이 글로벌 매출을 노린다는 뜻이고, 영어 커뮤니케이션이 실무다.
내 경험 MS M365 글로벌 지원 20명 중 1위(영어로 진단·문서화), 12시간 시차(시애틀) 비동기 협업 체계 설계. 영어·글로벌·비동기는 이미 내 실무 이력.

⑦ 예상 곤란 질문 — 정직 + 재프레임

답이 곤란한 질문일수록 연기하지 말고 사실을 인정한 뒤, 그 사실이 상대에게 손해가 아닌 축으로 옮긴다. 사실은 그대로 두고 의미의 무게중심만 이동시키는 것 — 자소서 정직 섹션과 같은 근육.

Q. "다른 Upstage 정규직 전형도 진행 중인 걸로 아는데, 둘 다 되면 어디로 가나요?"

전제 같은 회사 ATS라 그들은 이미 안다. "인턴도 똑같이 좋다"는 거짓말은 그 방에서 즉시 들통난다. 둘 다 되면 정규직 — 당연한 걸 부정하면 회피하는 사람이 된다. 인정하고 재프레임하면 성숙한 사람이 된다.
면접관이 진짜 걱정하는 것 (여기에 답한다) ① 우리 시간 낭비 아냐?   ② 인턴은 헤지라 대충 할 사람 아냐?   ③ 그래도 우리한테 남는 게 있나?

⑧ "AI 도구 어떻게 쓰나요?" — 최강 카드

왜 이게 결정적인가 이 보고서의 논지가 Upstage 이 팀의 문제 그 자체다. 제목이 "AI를 믿을 수 있게 쓰는 법" — = 계약서 추출에서 "AI 결과를 믿고 실무에 넣을 수 있는가". "제가 AI를 쓴다"가 아니라 "제가 이미 당신 팀의 문제를 AI로 풀어봤고, 그걸 글로 정리했습니다"가 된다.

① 컴포넌트 = 함수다. 화면 조각을 return하는 함수(네가 아는 그 함수). ② props = 함수 인자다. 부모가 자식 함수에 값을 넘기는 것. ③ 화면 = state의 함수다. 화면 = f(state) — 값이 바뀌면 React가 f를 다시 불러 화면을 새로 그린다. 막혀도 이 셋을 말하면 "구조는 안다"가 전달된다.

먼저 개념 4개 (외우지 말고 이해)

말진짜 뜻
컴포넌트화면 조각을 돌려주는 함수 하나. function UserList() { return (화면) }
state (useState)화면이 기억해야 하는 값. 이 값이 바뀌면 React가 화면을 자동으로 다시 그린다.
fetch인터넷 주소에서 데이터를 가져오기. 시간이 걸리니 await로 기다린다.
.map()데이터 목록을 화면 조각 목록으로 바꾸기. [data].map(x => <div>{x}</div>)
전체 그림 한 문장 "빈 목록으로 시작 → 버튼 누르면 fetch로 데이터를 채움 → state가 바뀌니 화면이 다시 그려짐 → map으로 목록을 카드로." 이 한 문장이 프론트 라이브 코딩의 80%.

가장 작은 실행 파일 (npm 없이, 저장 후 더블클릭)

warmup.html로 저장하면 브라우저에서 바로 돈다. React를 인터넷에서 불러오고 JSX를 즉석 변환한다 — 설치 0. 아래 라이브 데모로 실제 작동을 먼저 보라.

warmup.html · 핵심만 (주석을 읽으며 이해)
<div id="root"></div>
<script src="https://unpkg.com/react@18/umd/react.development.js"></script>
<script src="https://unpkg.com/react-dom@18/umd/react-dom.development.js"></script>
<script src="https://unpkg.com/@babel/standalone/babel.min.js"></script>
<script type="text/babel">
  const { useState } = React;

  function UserList() {
    // 1) 화면이 기억할 값 3개 (빈 목록 / 로딩중? / 에러?)
    const [users, setUsers]     = useState([]);
    const [loading, setLoading] = useState(false);
    const [error, setError]     = useState(null);

    // 2) 데이터를 가져오는 함수
    async function load() {
      setLoading(true); setError(null);      // 시작: 로딩 켜고 에러 지움
      try {
        const res  = await fetch("https://jsonplaceholder.typicode.com/users");
        if (!res.ok) throw new Error("HTTP " + res.status);  // 실패 체크
        const data = await res.json();       // 응답을 데이터로
        setUsers(data);                      // ← 여기서 state가 바뀌면 화면이 다시 그려진다
      } catch (e) {
        setError(e.message);
      } finally {
        setLoading(false);                   // 끝: 로딩 끔
      }
    }

    // 3) 화면 (state에 따라 셋 중 하나가 보인다)
    return (
      <div>
        <button onClick={load}>사용자 불러오기</button>
        {loading && <p>불러오는 중…</p>}
        {error   && <p>에러: {error}</p>}
        {users.map((u) => (                  // 목록 → 카드 목록
          <div key={u.id}>{u.name} — {u.email}</div>   // key 꼭!
        ))}
      </div>
    );
  }
  ReactDOM.createRoot(document.getElementById("root")).render(<UserList />);
</script>
