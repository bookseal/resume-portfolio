# 지원서 생성 파이프라인

이 repo 는 **이력서·자기소개서·포트폴리오 생성기**다. JD 라는 함수에 내 원자료를 통과시켜
조준된 지원서를 만든다. 아래 순서를 따르되, 각 단계의 **판단 규칙**을 반드시 지킨다.
규칙은 7건의 실제 지원에서 학습된 것이다 — 어기면 같은 실수를 반복한다.

## 0. 입력 확인

사용자가 채팅에 **JD 원문을 붙여넣는다**. 붙여넣지 않았으면 요청한다.
JD 에서 **회사·직무·공고번호(URL)** 를 뽑는다. 안 보이면 사용자에게 묻는다.
폴더/파일 식별자는 `<회사>_<직무약칭>` 또는 `<회사>_<공고번호>_<직무>` (기존 `02_function/*_JD_*.md` 네이밍과 일관되게).

## 1. JD 를 원문 그대로 보존

`02_function/<회사>_JD_<식별자>.md` 로 저장. **원문을 요약하지 말 것** — 나중에 포지셔닝할 때
"저 회사가 정확히 어떤 단어를 썼는지"가 곧 자소서에서 되돌려줄 키워드다. 형식:

```markdown
# <직무명>

**URL:** <공고 URL>

## 직무 상세
<원문 전체를 인용/그대로>

## 요구역량 정리 (원문에서 추출)
| 구분 | 내용 |   ← 필수사항 / 우대사항 / 자격요건을 표로

## 나중에 볼 포인트
- <이 JD 만의 차별점, 조준할 각도, 지원 시 주의점>
```

## 2. 자산 로드 (초안 쓰기 전 필수)

반드시 먼저 읽는다:
- `01_input/제출이력.md` — 어디에 무엇을 어떤 서사로 냈는지, 재사용 가능한 자산, 미완 항목
- `02_function/04_자산_인벤토리.md` — 동원 가능한 모든 증거
- **최근 제출본 2~3개**의 md (`03_output/submitted/<날짜>_*/이력서*.md`, `자기소개서*.md`) — 문장 재사용 출발점

경력 본문은 사실이라 재사용 가능하지만, **헤드라인·요약·기술스택 순서는 회사의 함수**다. 다시 쓴다.

## 3. 이력서 md 작성 → `03_output/drafts/<식별자>/이력서_이기찬_<식별자>_<YYYYMMDD_HHMM>.md`

파일명 끝 타임스탬프는 **생성 시각**(`date "+%Y%m%d_%H%M"`).

**규칙 (모두 지킬 것):**
- **요약 = JD 필수사항에 1:1.** 요약 3~4불릿이 JD 의 필수 역량과 정확히 대응해야 한다.
  채용자는 요약을 JD 와 나란히 놓고 대조한다. 우대사항을 맨 위에 두지 말 것(필수를 놓친 것처럼 보임).
- **헤드라인은 이 JD 전용.** 남의 회사 직무명(FDE, Solution Engineer 등) 금지.
- **프로젝트 배치는 직무에 따라.** 경력 중시 직무면 경력 위주, **인턴·경력 안 보는 직무면
  프로젝트를 경력 위로** 올린다("무슨 경력"이 아니라 "뭘 만들 줄 아나"를 묻기 때문).
- **자산이 이 JD 에 어떻게 대응하는지** 확인. 같은 프로젝트라도 JD 에 따라 강조가 다르다
  (예: llm-app-lab 이 SRE 엔 인프라, Agent 엔 eval 하네스, 인턴엔 UI/UX).

## 4. 자기소개서 md 작성 → 같은 폴더, `자기소개서_이기찬_<식별자>_<타임스탬프>.md`

- **1~3장(내가 무엇을 했는가)은 최근 자소서에서 재사용 가능. 4~5장(왜 이 회사인가)은 통째로 다시 쓴다.**
- JD 가 스스로 쓴 문장을 인용해 되돌려준다 (그 팀의 자기 인식 = "우리 말을 이해한 사람").
- 마지막 장은 **정직 섹션**: 약점(연차·경험 부족·기술 폭)을 **먼저** 밝힌다. 감추면 면접에서
  30초 만에 드러나 "부풀리는 사람"이 되고, 먼저 말하면 "자기 수준을 아는 사람"이 된다.
- 오버스펙 우려가 있는 자리(인턴 등)는 "왜 이 자리인가"를 선제적으로 방어.

## 5. 사실 검증 (초안 후, 빌드 전)

- `grep -i "upstage\|solar\|cognition\|<이전지원회사>"` 로 **남의 회사명 잔존** 확인 → 중립화.
- 추측한 기술은 쓰지 않는다. JD 에 "허위사실 시 영입 취소" 조항이 흔하다.
  확신 없으면 `gh repo list bookseal` / 포트폴리오(`~/workspace/`)를 grep 해 근거를 찾는다.
  (예: Munhaepang 을 Next.js 로 쓰면 안 됨 — 실제 HTML. 검증된 React+Vite 로.)
- 살아있는 링크(`status.bit-habit.com`, `infra.bit-habit.com`, 각 프로젝트)를 앞으로 꺼낸다 —
  검증 가능한 증거는 문장보다 세다.

## 6. docx 빌드 (한 번에 여기까지)

```bash
python3 02_function/tools/md2docx_template.py 03_output/drafts/<식별자>/이력서_….md
python3 02_function/tools/md2docx_template.py 03_output/drafts/<식별자>/자기소개서_….md
```

- 템플릿·하이퍼링크·서식은 **자동**이다(파일명에 "자기소개서" 있으면 자소서 서식 자동 선택).
  URL 은 md 에 평문(`bit-habit.com`)으로 써도 하이퍼링크가 자동 생성된다.
- 빌드 출력의 **`분량: 약 N줄 … ✅/⚠️`** 를 확인한다. **⚠️ 면 1페이지 초과** —
  리포트가 짚어주는 3줄 불릿부터 압축한다. **단락이 아니라 줄이 예산**이다(1페이지 ≈ 55줄).
  줄이는 순서: ① 3줄 불릿 압축 → ② JD 무관 경력(오래된·타분야) → ③ 요약 불릿 1줄화 →
  기술스택은 **지우지 말 것**(ATS·리크루터 키워드 필터 대비).

## 7. 인계 (사용자가 마무리)

빌드 후 사용자에게 전달한다:
- **포트폴리오 조준 링크**: `https://bit-habit.com/?title=<직무명 URL 인코딩>` (필요시 `&focus=infra|solutions|edu`).
  공고 직함을 그대로 넣으면 히어로가 그 직함으로 뜬다.
- **1페이지 체크리스트**: Word 에서 열어 1페이지 확인 → **PDF 저장**(사용자가 직접).
- **⚠️ 사실 확인 필요 항목**(추측으로 넣은 게 있으면 명시).

## 8. 제출 후 (사용자가 "제출했다" 하면)

1. `03_output/drafts/<식별자>` → `03_output/submitted/<YYYY-MM-DD>_<식별자>/` (git mv). Word 잠금파일 `~$*` 제거.
2. 사용자가 Word 에서 손본 제출본(docx)이 있으면, **그 판을 템플릿으로 승격할지** 확인
   (서식만 바꿨으면 `02_function/templates/` 로 복사, 구조도 바꿨으면 md 문법에도 반영).
3. `SUBMITTED.md` 작성: 회사/직무/공고번호/**제출 시각(실제 발송 시각)**/결과/전형 + 포지셔닝 축 +
   JD 필수↔근거 매핑 + 재사용 자산 + 미완 항목.
4. `03_output/submitted/INDEX.md` 에 한 줄 추가.
5. `01_input/제출이력.md` 갱신.

## 파일명·경로 규칙

- 폴더 식별자는 영어, 제출 폴더는 `YYYY-MM-DD_회사_직무`.
- 파일명 끝 `_YYYYMMDD_HHMM` 은 **생성 시각**(`date` 로 찍는다). 새 버전 = 새 타임스탬프.
- **md 가 원본이다.** Word 에서 문장을 고쳤으면 md 에도 반영. 서식만 손봤으면 재빌드하지 말 것(날아간다).

## 하지 않는 것

- **사전질문 자동 생성 안 함.** Upstage 류의 "제품 평가·1000자 경험"은 사용자가 따로 요청할 때만.
  (요청 시: 실제 사용 경험만 쓰고 미평가는 정직하게 명시. Document Parse/IE 는 아직 미평가 상태.)
- PDF 변환·메일 발송은 사용자가 직접(Word 에서 PDF, 메일은 본인이 발송).
